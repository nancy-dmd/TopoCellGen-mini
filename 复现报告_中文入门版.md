# TopoCellGen 复现报告（中文入门版）

## 1. 这是什么项目

`TopoCellGen` 是一个用于生成病理细胞布局（cell layout）的扩散模型项目。

它的输入不是一张普通图像，而是一个三通道的细胞点图：

- 第 1 个通道表示一类细胞
- 第 2 个通道表示一类细胞
- 第 3 个通道表示一类细胞

模型的目标不是“把一张图修复成另一张图”，而是：

- 在给定细胞数量条件的情况下
- 生成一张新的、合理的细胞空间分布图

所以这个项目更像是“条件生成”，而不是“分割”或“检测”。

## 2. 这次复现做了什么

这次复现没有训练模型，而是走了最小可运行路线：

1. 下载官方代码仓库
2. 下载作者提供的预训练权重 `brca_m2c.pt`
3. 从 BRCA-M2C 数据集中只取 1 个测试样本
4. 把原始标注转换成项目需要的 `256x256x3 .npy` 格式
5. 用预训练模型生成 1 个细胞布局结果
6. 保存生成结果、计数对比和可视化图

这是一种适合初学者的复现方式，因为：

- 不需要训练
- 不需要大量 GPU
- 不需要完整数据集
- 能最快验证“项目是否真的跑通”

## 4. 数据是怎么准备的

原始 BRCA-M2C 数据集里的标注是文本格式，每一行表示一个细胞点：

```text
<Y坐标> <X坐标> <细胞类别ID>
```

例如：

```text
178 376 1
180 128 1
2 349 2
```

这里的类别 ID 含义是：

- `1` = Lymphocytes
- `2` = Tumor/Epithelial
- `3` = Stromal

为了让 TopoCellGen 能直接读取，这些点被转换成了一个 `256x256x3` 的 numpy 数组：

- 高和宽是 `256`
- 通道数是 `3`
- 每个细胞点在对应通道上被写成一个小的 `3x3` 方块

这样模型就能像读取图像一样读取细胞布局。

本次只准备了 1 个最小测试样本：

- `data/brca_minimal/test_dataset/TCGA-A8-A099-01Z-00-DX1_29501_22501_1000_1000_0.93.npy`

## 5. 做了哪些代码修改

为了让项目更容易复现，这次主要做了三类修改。

### 5.1 生成脚本参数化

原始 `generate_layout_brca.py` 里有很多硬编码路径，比如：

- 模型路径
- 测试数据路径
- 输出结果路径

这会导致代码在别的机器上几乎无法直接运行。

现在已经改成可以通过命令行参数指定：

- `--model_path`
- `--test_patch_path`
- `--results_root_path`
- `--max_test_files`

这样就更适合复现和教学。

### 5.2 增加最小数据准备脚本

新增脚本：

- `tools/prepare_brca_minimal.py`

作用是把 BRCA-M2C 的原始标签文本，转换成 TopoCellGen 需要的 `.npy` 点图格式。

这一步对初学者很重要，因为原仓库没有直接提供“从原始标签到推理输入”的最小脚本。

### 5.3 让推理不再依赖训练专用拓扑损失包

原项目在导入扩散模块时，会顺带导入训练时才需要的拓扑损失模块。

这会带来一个问题：

- 即使你只是想做推理
- 也必须安装一串复杂的拓扑依赖

这对入门者非常不友好。

本次修改把这部分处理成了“可选依赖”：

- 训练时仍然可以使用
- 推理时如果不用这部分 loss，就不会阻塞生成

这样最小复现链路更稳。

## 6. 实际运行命令

### 6.1 准备最小数据

```bash
python tools/prepare_brca_minimal.py \
  --labels-dir ../Dataset-BRCA-M2C/labels \
  --split-file ../Dataset-BRCA-M2C/brca_ds_test.txt \
  --output-dir data/brca_minimal/test_dataset \
  --limit 1
```

### 6.2 运行生成

```bash
python generate_layout_brca.py \
  --model_path checkpoints/brca_m2c.pt \
  --test_patch_path data/brca_minimal/test_dataset \
  --results_root_path results_minimal \
  --max_test_files 1
```

## 8. 结果应该怎么理解

这是初学者最容易误解的一部分。

### 8.1 不要求和 GT 逐像素一致

这个项目是“生成模型”，不是“重建模型”。

所以生成结果不应该理解成：

- 必须和 GT 每个位置都完全重合
- 必须每个红点、绿点、蓝点都出现在同一个坐标

它真正追求的是：

- 细胞总数接近
- 各类别数量接近
- 空间拓扑结构合理
- 局部聚集和分散模式与真实数据相似

### 8.2 这次结果如何

本次样本的计数对比如下：

- Ground Truth: `[11, 77, 27]`
- Generated: `[10, 76, 28]`

总数对比：

- Ground Truth 总数：`115`
- Generated 总数：`114`

误差：

- 总误差：`1`
- 每类误差：`[1, 1, 1]`

对于一个单样本的最小复现来说，这个结果已经说明：

- 生成主流程是正常的
- 预训练模型能够读取条件并生成合理布局
- 后处理与保存链路也是通的

### 8.3 那张对比图怎么看

对比图文件：

- `results_minimal/2026-05-02/15-26-04/comparison_gt_vs_generated.png`

看图时建议重点看：

1. 三类细胞的大致数量是否接近
2. 颜色对应的空间分布是否合理
3. 有没有明显不自然的空洞、团块或异常噪声

不要按“分割结果叠加”的思路去看，因为这里不是逐点匹配任务。

## 9. 这次复现解决了哪些常见坑

### 9.1 路径硬编码

原代码默认写死路径，换机器后很容易直接报错。

已经通过参数化解决。

### 9.2 推理被训练依赖卡住

原代码导入链会拉起拓扑 loss 相关依赖，导致即使只是生成，也可能报模块缺失。

这次做了推理友好的兼容处理。

### 9.3 权重文件过大

`brca_m2c.pt` 体积约 `2.1GB`，超过 GitHub LFS 单文件 `2GB` 上限。

因此：

- 代码、最小数据、结果可以上传到仓库
- 但 checkpoint 更适合通过外链方式提供

这也是很多大模型项目常见的现实问题。

## 10. 对入门者的建议

如果你是第一次复现这种项目，建议按下面顺序理解：

1. 先跑通最小推理
2. 再理解输入 `.npy` 的三通道表示方式
3. 再看生成结果和 GT 的计数关系
4. 最后再去研究拓扑 loss、评估指标和训练逻辑

不要一开始就陷进完整训练流程，因为：

- 依赖重
- 数据准备复杂
- 训练成本高
- 失败点太多

先证明“项目能跑”，再逐步理解“它为什么这样设计”，学习效率会更高。
